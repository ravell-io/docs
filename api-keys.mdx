## What are API keys?

**API keys** let you access Ravell programmatically through the [Knowledge API](/knowledge-api). Use them to search your connected sources, query entities, or generate answers and digests from external tools, scripts, or integrations.

API keys are workspace-scoped — each key belongs to a specific workspace and can only access data within that workspace.

---

## Create an API key

<Steps>
  <Step title="Open settings">
    Go to **Settings → API Keys**.
  </Step>
  <Step title="Create a new key">
    Click **Create API key**. Enter a name (e.g. "CI pipeline" or "Dashboard integration") and select the scopes you need.
  </Step>
  <Step title="Copy the token">
    The full token is shown only once. Copy it and store it securely. You won't be able to see it again.
  </Step>
</Steps>

<Warning>
  Treat API keys like passwords. Never commit them to source control or share them in plaintext.
</Warning>

---

## Scopes

Each API key has one or more scopes that control what it can access:

| Scope | Access level | Endpoints |
|-------|-------------|-----------|
| `read` | Read-only | Search, entities, related |
| `read_write` | Read and write | Everything in `read`, plus query and digest |

A `read_write` key automatically covers all `read` operations. Choose `read` if you only need search and lookup functionality.

---

## Token format

API key tokens follow the format `ravelk_` followed by 48 hex characters. When authenticating, pass the token as a Bearer token in the `Authorization` header:

```bash
curl -H "Authorization: Bearer ravelk_your_token_here" \
  https://app.ravell.io/api/knowledge/search?query=churn+signals
```

---

## Expiration and revocation

- **Expiration**: You can optionally set an expiration date when creating a key. Expired keys stop working automatically.
- **Revocation**: To immediately disable a key, go to **Settings → API Keys** and revoke it. Revoked keys cannot be restored.

---

## Best practices

- **Use descriptive names** so you can identify what each key is used for.
- **Set expirations** for keys used in temporary or experimental contexts.
- **Use the minimum scope** needed — prefer `read` unless you need query or digest access.
- **Rotate keys periodically** by creating a new key and revoking the old one.
- **Never expose keys client-side** — API keys should only be used in server-side code or secure environments.
